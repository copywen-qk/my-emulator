# NEMU-RV32: Control Flow (JAL, BNE)

## [韌體筆記]
### 打亂的立即數 (Scrambled Immediates)
在 RISC-V 的 B-Type 與 J-Type 指令中，立即數的位元並非按順序排列，而是被打亂後放置在指令中（例如 B-Type 的 `12|10:5|4:1|11`）。
- **原因：** 這是為了讓所有指令格式（R/I/S/B/U/J）中，相同的暫存器欄位（如 `rs1`, `rs2`, `rd`）盡可能保持在相同的位置。這簡化了硬體中多工器（Multiplexer）的設計，降低了邏輯門的數量與功耗。
- **next_pc 的防護：** 在模擬器實作中，我們使用 `next_pc` 變數先計算預設的 `pc + 4`。如果指令是跳躍指令且條件成立，再覆蓋 `next_pc`。最後統一更新 `cpu.pc = next_pc`。這樣可以避免在執行指令過程中直接修改 `cpu.pc` 導致偏移計算錯誤（例如 `pc = pc + offset` 時，若 `pc` 已經先被 `+4` 就會出錯）。

## [CSAPP 筆記]
### 控制危險 (Control Hazard) 與分支預測
- **Control Hazard：** 當處理器處於流水線（Pipeline）模式時，下一條指令在當前指令解碼完成前就已經被取指。如果當前指令是跳躍指令，則預先取的指令可能是錯誤的，這稱為控制危險。
- **分支預測 (Branch Prediction)：** 現代 CPU 會預測分支是否跳轉（Taken or Not Taken）。
  - **靜態預測：** 例如「向後跳轉通常是迴圈，預測為 Taken」。
  - **動態預測：** 使用歷史表紀錄過去的行為。
- **效能影響：** 預測失敗會導致流水線沖刷（Pipeline Flush），損失多個時鐘週期。雖然我們的模擬器是單週期模擬，不涉及危險，但理解這些概念對優化真實硬體上的程式至關重要。
